Synchronous

Blocking operations - Each operation must complete before the next begins

Task 1 (5s)
Task 2 (3s)
Task 3 (2s)

Total time: 10 seconds

Asynchronous

Non-blocking operations - Operations can run in parallel

Task 1 (5s)
Task 2 (3s)
Task 3 (2s)

Total time: ~5 seconds

Node.js uses an event-driven, non-blocking I/O model that makes it lightweight and efficient.

Node.js Module System

Exporting from a Module

In Node.js, each file is treated as a separate module. To make functions, objects, or values available to other files, you need to export them.

// mathUtils.js
const add = (a, b) => a + b;
const subtract = (a, b) => a - b;

// Exporting multiple functions
module.exports = {
  add,
  subtract
};

Importing a Module

To use functionality from another module, you need to require it.

// app.js
const math = require('./mathUtils');

console.log(math.add(5, 3)); // 8
console.log(math.subtract(5, 3)); // 2

// Or with destructuring
const { add, subtract } = require('./mathUtils');

Module Types

Local Modules

Files you create in your project

const myModule = require('./myModule');

Core Modules

Built-in Node.js modules

const fs = require('fs');

NPM Modules

Third-party packages from npm

const express = require('express');

File System (fs) Module

The fs module allows you to work with the file system on your computer. It provides both synchronous and asynchronous methods.

Common fs Operations

Reading Files

// Asynchronous
fs.readFile('file.txt', 'utf8', (err, data) => {
  if (err) throw err;
  console.log(data);
});

// Synchronous
const data = fs.readFileSync('file.txt', 'utf8');

Writing Files

// Asynchronous
fs.writeFile('file.txt', 'Hello!', (err) => {
  if (err) throw err;
  console.log('File saved!');
});

// Synchronous
fs.writeFileSync('file.txt', 'Hello!');

Other Operations

// Rename file
fs.rename('old.txt', 'new.txt', cb);

// Delete file
fs.unlink('file.txt', cb);

// Create directory
fs.mkdir('folder', cb);

When to Use Sync vs Async

Synchronous

  • Blocking operations
  • Simpler code flow
  • Use for scripts, not servers

Asynchronous

  • Non-blocking operations
  • Better for servers
  • Requires callbacks or promises

Events & EventEmitter

Event-Driven Architecture

Node.js is built around an event-driven architecture where objects (emitters) emit named events that cause function objects (listeners) to be called.

const EventEmitter = require('events');
const myEmitter = new EventEmitter();

// Register a listener
myEmitter.on('event', () => {
  console.log('An event occurred!');
});

// Emit the event
myEmitter.emit('event');
EventEmitter
.on('event', listener)
.emit('event')
Listener Function

Common EventEmitter Methods

.on(event, listener)

Adds a listener function for the specified event

emitter.on('data', (data) => {
  console.log(data);
});

.emit(event, [...args])

Calls each listener registered for the event

emitter.emit('data', { id: 1 });

.once(event, listener)

Adds a one-time listener that's removed after being called

emitter.once('connect', () => {
  console.log('Connected!');
});

package-lock.json

The package-lock.json file is automatically generated when you install packages with npm. It records the exact version of every installed package and its dependencies.

Why package-lock.json is Important

Version Locking

Ensures that the exact same versions of dependencies are installed on all environments (development, production, CI)

Dependency Tree

Records the complete tree of dependencies, including nested dependencies

Installation Speed

Allows npm to skip version resolution for faster installs

Key Differences

package.json

  • Lists direct dependencies
  • Uses semantic versioning (^, ~)
  • Manually edited
  • Lists both dependencies and devDependencies

package-lock.json

  • Lists exact versions of all dependencies
  • No semantic versioning - exact versions only
  • Automatically generated
  • Includes integrity checks (hashes)

Best Practices

  • Commit to version control: Always include package-lock.json in your repository
  • Don't edit manually: Let npm manage this file
  • Regenerate when needed: Delete and run npm install if you suspect issues

HTTP Module

The http module allows Node.js to transfer data over HTTP. It can create an HTTP server that listens to server ports and gives a response back to the client.

Creating a Basic HTTP Server

const http = require('http');

const server = http.createServer((req, res) => {
  // Handle request
  res.statusCode = 200;
  res.setHeader('Content-Type', 'text/plain');
  res.end('Hello World!');
});

server.listen(3000, () => {
  console.log('Server running on port 3000');
});

Key Components

http.createServer()

Creates a new HTTP server instance

Takes a request listener function that's called for each request

Request Object (req)

Contains information about the incoming request:

  • req.url - Request URL
  • req.method - HTTP method
  • req.headers - Request headers

Response Object (res)

Used to send data back to the client:

  • res.statusCode - Set status code
  • res.setHeader() - Set response headers
  • res.end() - Send response and end connection

Handling Different Routes

const server = http.createServer((req, res) => {
  if (req.url === '/') {
    res.end('Home page');
  } else if (req.url === '/about') {
    res.end('About page');
  } else {
    res.statusCode = 404;
    res.end('Page not found');
  }
});

Basic Routing

Routing refers to how an application's endpoints (URIs) respond to client requests. In Node.js without Express, you handle routing manually by examining the request URL and method.

Basic Routing Example

const http = require('http');

const server = http.createServer((req, res) => {
  // Extract URL and method
  const { url, method } = req;

  // Home page
  if (url === '/' && method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'text/html' });
    res.end('<h1>Home Page</h1>');
  }

  // About page
  else if (url === '/about' && method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'text/html' });
    res.end('<h1>About Page</h1>');
  }

  // API endpoint
  else if (url === '/api/users' && method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify([{ id: 1, name: 'John' }]));
  }

  // Not found
  else {
    res.writeHead(404, { 'Content-Type': 'text/html' });
    res.end('<h1>Page Not Found</h1>');
  }
});

server.listen(3000);

Routing Challenges

Manual Parsing

You need to manually parse URLs and query parameters

const url = require('url');
const parsedUrl = url.parse(req.url, true);
const path = parsedUrl.pathname;
const query = parsedUrl.query;

Complexity

Routing becomes complex with many endpoints

Large if-else or switch statements

No built-in support for route parameters

Middleware

No built-in middleware support

Need to manually handle things like:

  • JSON parsing
  • Authentication
  • Error handling

Why Use Express for Routing?

Express provides a much cleaner and more powerful routing system:

// With Express
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send('Home Page');
});

app.get('/about', (req, res) => {
  res.send('About Page');
});

app.get('/api/users', (req, res) => {
  res.json([{ id: 1, name: 'John' }]);
});

app.listen(3000);

Express Middleware

Middleware functions are functions that have access to the request object (req), the response object (res), and the next middleware function in the application's request-response cycle.

Middleware Pipeline

Request
Middleware 1
Middleware 2
Route Handler
Response

Types of Middleware

Application-level

app.use((req, res, next) => {
  console.log('Time:', Date.now());
  next();
});

Router-level

router.use((req, res, next) => {
  console.log('Router middleware');
  next();
});

Error-handling

app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).send('Error!');
});

Built-in

app.use(express.json());
app.use(express.urlencoded());
app.use(express.static('public'));

Common Middleware Examples

// Logger middleware
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next();
});

// Authentication middleware
const auth = (req, res, next) => {
  if (req.headers.authorization) {
    next();
  } else {
    res.status(401).send('Unauthorized');
  }
};

// Apply to specific route
app.get('/protected', auth, (req, res) => {
  res.send('Protected content');
});

REST & CRUD with Express

GET POST PUT DELETE

REST (Representational State Transfer) is an architectural style for designing networked applications. CRUD (Create, Read, Update, Delete) are the basic operations of persistent storage.

RESTful Routes for a Resource

HTTP Method Route Description CRUD
GET /users Get all users Read
GET /users/:id Get single user Read
POST /users Create new user Create
PUT /users/:id Update entire user Update
PATCH /users/:id Partially update user Update
DELETE /users/:id Delete user Delete

Express CRUD Implementation

const express = require('express');
const app = express();
app.use(express.json());

let users = [{ id: 1, name: 'John' }];

// GET all users
app.get('/users', (req, res) => {
  res.json(users);
});

// GET single user
app.get('/users/:id', (req, res) => {
  const user = users.find(u => u.id === parseInt(req.params.id));
  if (!user) return res.status(404).send('User not found');
  res.json(user);
});

// POST create user
app.post('/users', (req, res) => {
  const user = {
    id: users.length + 1,
    name: req.body.name
  };
  users.push(user);
  res.status(201).json(user);
});

// PUT update user
app.put('/users/:id', (req, res) => {
  const user = users.find(u => u.id === parseInt(req.params.id));
  if (!user) return res.status(404).send('User not found');

  user.name = req.body.name;
  res.json(user);
});

// DELETE user
app.delete('/users/:id', (req, res) => {
  const userIndex = users.findIndex(u => u.id === parseInt(req.params.id));
  if (userIndex === -1) return res.status(404).send('User not found');

  users.splice(userIndex, 1);
  res.status(204).send();
});

app.listen(3000);

Best Practices

  • Use proper HTTP methods: GET for read, POST for create, PUT/PATCH for update, DELETE for delete
  • Use appropriate status codes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 404 Not Found, 500 Server Error
  • Consistent naming: Use plural nouns for resources (/users, /products)
  • Version your API: Consider prefixing routes with /api/v1/
  • Error handling: Return meaningful error messages with proper status codes
